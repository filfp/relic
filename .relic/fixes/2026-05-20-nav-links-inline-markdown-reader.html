Fix: nav links open raw markdown instead of rendering inline

ID: 2026-05-20-nav-links-inline-markdown-reader Date: 2026-05-20 Spec: 008-html-spec-mode Status: solved Classification: misspecification

Issue

The sticky header in spec HTML files includes nav links to spec.md, plan.md, and tasks.md. Clicking any of these links causes the browser to navigate to the raw .md file and render it as plain text — displaying the full Markdown syntax (asterisks, hashes, brackets) with no formatting. This is worse than having no links at all: the experience is disorienting and provides zero readable content.

Impact: Every spec HTML file in html mode has non-functional nav links. Clicking any header nav link forces the user out of the styled HTML view into a raw text dump. There is no back button — the user must use the browser's back button to return, losing scroll position and dark mode state.

The expected behaviour: clicking a nav link should render the markdown content inline within the same page, preserving the sticky header (with a back button), applying the same design system (CSS custom properties, dark mode), and providing a polished reading experience with proper heading hierarchy, code blocks, and formatting.

Root Cause

misspecification — FR-24 specifies "navigation links to spec.md / plan.md / tasks.md as relative paths" without specifying the rendering experience. The implementation faithfully follows this (plain <a href="spec.md"> tags), but the natural browser behaviour — navigating to a .md file and displaying raw text — is unacceptable as a user experience. The spec must be amended to require inline markdown rendering.

Why not code-bug?

The implementation is correct given what FR-24 says. The spec did not specify how the linked files should be displayed — it only said "relative paths". The gap is in the spec, not in the code that follows it.

Technical constraint: file:// CORS

The obvious fix — fetch() the .md file on click — is blocked in Chrome and most browsers when the HTML file is opened via file:// (same-origin CORS restriction on local files). Firefox allows it; Chrome does not. This constraint eliminates a pure runtime fetch() approach as the primary mechanism.

The correct solution uses a two-layer approach:

  1. Primary: Embed markdown source in the HTML as inert <script type="text/plain"> blocks. The LLM populates these during the HTML step of each workflow command. Zero fetch, zero CORS — works everywhere.
  2. Fallback: If an embedded source block is empty (not yet populated), try fetch() — works on HTTP-served files and Firefox file://. If that also fails, show a graceful message with an "Open in new tab" escape hatch.

Proposed Changes

1 — templates/base.html — inline markdown reader

Three additions to the base template:

  • Three <script type="text/plain"> source blocks with IDs relic-src-spec, relic-src-plan, relic-src-tasks. The LLM fills these during the HTML step. Initially empty.
  • A lightweight inline markdown-to-HTML parser (~120 lines of JS, no CDN). Handles: # H1–H4, **bold**, *italic*, `code`, ` ```fenced blocks``` `, - lists, 1. ordered lists, > blockquote, [text](url), --- rules, paragraphs.
  • A markdown reader panel renderer + back button logic:
    • Nav link clicks are intercepted (preventDefault).
    • Content is read from the embedded source block first; if empty, falls back to fetch().
    • A reader panel replaces #relic-body content with the rendered markdown.
    • A ← back button appears in the header (replacing the nav links); clicking it restores the original HTML content.
    • The panel applies the design system CSS variables (var(--text), var(--surface), etc.) for full dark mode compatibility.

2 — All 7 prompt templates — HTML step update

Add to the HTML step instruction in specify.md, clarify.md, plan.md, tasks.md, implement.md, fix.md, solve.md:

New instruction (verbatim): After writing the HTML content sections, update the three source blocks with the current file content: <script type="text/plain" id="relic-src-spec"> ← current spec.md, <script type="text/plain" id="relic-src-plan"> ← current plan.md, <script type="text/plain" id="relic-src-tasks"> ← current tasks.md. These blocks power the inline markdown reader — keep them current.

For fix HTML files (fix.md, solve.md): the embedded source blocks reference the fix document itself and any relevant spec files, not the standard spec/plan/tasks trio.

3 — shared/contracts/HtmlComponentContract.md — reader contract

Add a new section documenting the inline markdown reader:

  • Source block IDs and their purpose
  • Parser behaviour (elements supported)
  • Back navigation contract (back button replaces nav links in header when reader is open)
  • Fallback behaviour (fetch → graceful message)
  • LLM obligation: keep source blocks populated during every HTML step

4 — specs/008-html-spec-mode/spec.md — amend FR-24

Change "navigation links to spec.md / plan.md / tasks.md as relative paths" to specify inline rendering: "clicking a nav link renders the linked markdown file inline within the same page using the embedded markdown reader; a back button in the header returns to the spec HTML view."
